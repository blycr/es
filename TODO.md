# 待办事项

本文件用于记录已经形成方向、但尚未实现的功能规划。

## 优先规划：AI 智能化编辑

### 目标

在保持本地优先使用方式的前提下，为 `.ass` 字幕文件增加可选的 AI 辅助编辑能力。

AI 功能应当作为增强能力存在，而不是替代现有的本地可视化编辑和原文编辑。

### 设计原则

- 默认关闭，只有用户主动配置后才可用
- 默认最小化上传内容，避免整份字幕直接发送到远程
- 预设与自定义都允许用户继续修改参数
- 提供“建议预览”或“变更预览”，避免模型直接静默改写全部内容
- 不把 API Key 上传到项目服务器，优先保存在本地环境

### 第一阶段能力范围

- 基于用户指令生成字幕修改建议
- 针对 `Script Info`、`Aegisub Project Garbage`、`V4+ Styles`、`Events` 生成结构化修改结果
- 对文件名、样式名、事件文本、元信息等常见字段提供批量修改建议
- 支持基于选中文件、选中区块、选中行进行局部 AI 处理

### 提供商方案

#### 预设提供商

首批预设建议提供以下类型：

- OpenAI
- Anthropic
- Google Gemini
- DeepSeek
- 阿里云百炼
- 火山方舟

说明：

- 预设的意义是减少首次配置成本，不是锁死配置
- 进入任一预设后，用户仍可修改 Base URL、模型名、请求参数、系统提示词和用户提示词模板

#### 自定义提供商

除预设外，还应提供“自定义 API”模式，至少支持以下字段：

- 提供商名称
- Base URL
- 接口路径
- 鉴权方式
- 模型名称
- 请求头扩展
- 系统提示词模板
- 用户提示词模板
- 温度、最大输出长度等参数

### 配置模型

建议把 AI 配置拆成两层：

#### 提供商配置

- API Key
- Base URL
- 模型名称
- 默认请求参数

#### 任务配置

- 任务名称
- 作用范围
- 提示词模板
- 输出格式要求
- 是否允许覆盖原内容

这样用户既可以维护一组固定提供商，也可以在不同字幕任务之间复用不同提示词方案。

### 上下文与安全策略

为了控制成本、上下文长度和隐私风险，建议采用以下策略：

- 默认只发送结构摘要、字段列表、样本片段和用户当前选中的内容
- 对 `Events` 采用分块处理，而不是整表直接发送
- 对超长字幕按文件、区块、页或筛选结果分批处理
- 每次请求前显示“将发送哪些内容”的摘要
- 提供“仅生成建议，不直接应用”的安全模式

### 交互流程建议

1. 用户启用 AI 功能
2. 选择预设提供商或自定义提供商
3. 填入 API Key，并按需修改 Base URL、模型和提示词
4. 选择处理范围：整文件、当前区块、当前页、选中条目
5. 生成修改建议
6. 在本地预览修改前后差异
7. 用户确认后再应用到字幕内容

### UI 规划建议

- 在现有“直接编辑”区域增加单独的 AI 工具入口，而不是打断当前编辑流
- AI 配置区域支持折叠，避免常态下占用主界面空间
- 提供“最近使用的提供商配置”和“最近使用的提示词方案”
- 对敏感字段如 API Key 提供显示/隐藏和清空功能

### 数据存储建议

- API 配置默认仅保存在浏览器本地
- 支持“只保存在当前会话”与“保存在本地浏览器”两种模式
- 不建议在未加密的情况下导出包含 API Key 的完整配置

### 输出格式建议

AI 返回结果建议优先使用结构化格式，而不是纯文本描述，例如：

- 键值更新列表
- 样式字段更新列表
- 事件行修改列表
- 文件名修改列表

这样更容易做本地校验、预览和回滚。

### 风险与限制

- 不同提供商接口格式并不一致，需要做适配层
- 模型输出可能不稳定，必须有本地校验与回退机制
- 对超长字幕必须做严格的分块和限流处理
- 不应让 AI 请求阻塞现有本地编辑主流程

### 建议实施顺序

1. 先做提供商配置层与本地存储
2. 再做 OpenAI 兼容协议的通用接入
3. 在通用层跑通后，再补各大厂预设
4. 最后增加提示词模板管理、差异预览和批量应用

### 暂不建议在第一阶段做的内容

- 自动为所有文件整仓无确认改写
- 完全自由格式的模型输出直接写回字幕
- 把 AI 功能放到现有编辑流程的默认入口
